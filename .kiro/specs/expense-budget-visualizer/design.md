# Design Document: Expense & Budget Visualizer

## Overview

The Expense & Budget Visualizer is a single-page, client-side web application built with plain HTML, CSS, and Vanilla JavaScript. There is no build step, no backend, and no framework — the entire app ships as static files deployable from any web server or opened directly from the filesystem.

The core data flow is simple:
1. User submits the Input_Form → validate → create Transaction → update in-memory array → persist to LocalStorage → re-render UI (Balance_Display, Transaction_List, Chart).
2. User clicks delete → remove from in-memory array → persist to LocalStorage → re-render UI.
3. On page load → read LocalStorage → hydrate in-memory array → render UI.

Chart.js (v4.x) is loaded from the jsDelivr CDN and provides the pie chart rendering. All other logic is hand-written JavaScript.

---

## Architecture

The app follows a **unidirectional data flow** pattern without a framework:

```
User Action
    │
    ▼
Input Handler (in app.js)
    │
    ▼
Validator (validateForm / validateTransaction)
    │
    ▼
State Mutation (transactions[] array)
    │
    ├──► Storage Module (LocalStorage read/write)
    │
    └──► Renderer (updateBalanceDisplay, renderList, updateChart)
```

### File Structure

```
/
├── index.html          # Single HTML page, all semantic markup
├── css/
│   └── style.css       # All styles; responsive, WCAG AA compliant
└── js/
    └── app.js          # All application logic (state, validation, storage, rendering)
```

Chart.js is loaded via CDN `<script>` tag in `index.html`, before `app.js`.

### Module Boundaries (within app.js)

`app.js` is organized into logical sections with clear comments. There is no module bundler, so everything lives in one file under an IIFE or with careful variable scoping to avoid polluting the global namespace:

| Section | Responsibility |
|---|---|
| Constants | Category definitions, LocalStorage key, validation limits |
| State | The single `transactions` array that is the source of truth |
| Validator | Pure functions that check field values and return error messages |
| Storage | `loadTransactions()`, `saveTransactions()`, error handling |
| Renderer | `renderList()`, `updateBalance()`, `updateChart()` |
| Event Handlers | Form submit, delete button delegation, DOMContentLoaded init |

## Components and Interfaces

### 1. Input_Form

**HTML elements:**
- `<input type="text" id="item-name" maxlength="100">` — item name field
- `<input type="number" id="amount" min="0.01" max="999999999.99" step="0.01">` — amount field
- `<select id="category">` with options: `<option value="">-- Select --</option>`, `<option value="Food">Food</option>`, `<option value="Transport">Transport</option>`, `<option value="Fun">Fun</option>`
- `<button type="submit">Add Transaction</button>`
- Inline error `<span>` elements adjacent to each field for validation messages

**JavaScript interface:**

```js
// Returns null if valid, or an object mapping fieldName → errorMessage
function validateForm(name, amount, category) { ... }

// Called on form 'submit' event
function handleFormSubmit(event) { ... }
```

### 2. Transaction Data Model

```js
// Transaction object shape
{
  id: string,        // crypto.randomUUID() or Date.now().toString() fallback
  name: string,      // 1–100 characters, trimmed
  amount: number,    // float, 0.01 – 999,999,999.99
  category: string,  // 'Food' | 'Transport' | 'Fun'
  createdAt: number  // Date.now() timestamp for ordering
}
```

### 3. State

```js
// Single source of truth — mutated by add/delete operations
let transactions = [];  // Transaction[]
```

### 4. Balance_Display

**HTML:** `<span id="balance-value">$0.00</span>` inside a fixed header element.

**JavaScript interface:**

```js
// Recomputes sum of all transaction amounts and updates the DOM element
function updateBalance() { ... }
```

### 5. Transaction_List

**HTML:** `<ul id="transaction-list">` with `<li>` items generated by the renderer.

Each list item structure:
```html
<li data-id="{transaction.id}">
  <span class="tx-name">{name}</span>
  <span class="tx-category">{category}</span>
  <span class="tx-amount">{formatted amount}</span>
  <button class="delete-btn" aria-label="Delete {name}">×</button>
</li>
```

**JavaScript interface:**

```js
// Clears and re-renders the entire list from the transactions array
function renderList() { ... }

// Formats a number as currency string: "$1,234.56"
function formatCurrency(amount) { ... }

// Event delegation on <ul>: catches clicks on .delete-btn children
function handleDeleteClick(event) { ... }
```

### 6. Chart (Pie Chart via Chart.js)

**HTML:** `<canvas id="spending-chart"></canvas>` wrapped in a `<div id="chart-container">`. An empty-state `<p id="chart-empty-msg">` sits alongside the canvas.

**JavaScript interface:**

```js
// Chart.js instance (module-level variable)
let chartInstance = null;

// Aggregates amounts by category and updates or creates the Chart.js instance
function updateChart() { ... }

// Called once on init to set up the Chart.js pie chart skeleton
function initChart() { ... }
```

**Chart.js configuration:**

```js
{
  type: 'pie',
  data: {
    labels: ['Food', 'Transport', 'Fun'],
    datasets: [{
      data: [foodTotal, transportTotal, funTotal],
      backgroundColor: ['#FF6384', '#36A2EB', '#FFCE56'],
      hoverOffset: 4
    }]
  },
  options: {
    responsive: true,
    plugins: {
      legend: { position: 'bottom' },
      tooltip: { enabled: true }
    }
  }
}
```

Category colors are fixed constants so they never change across renders:
- Food: `#FF6384` (rose)
- Transport: `#36A2EB` (blue)
- Fun: `#FFCE56` (yellow)

### 7. Storage Module

**JavaScript interface:**

```js
const STORAGE_KEY = 'expense_visualizer_transactions';

// Reads and parses LocalStorage; returns Transaction[] or []
// Discards malformed data and displays an error notification
function loadTransactions() { ... }

// Serializes transactions[] to JSON and writes to LocalStorage
// On failure, shows a persistent inline error without reverting in-memory state
function saveTransactions() { ... }
```

### 8. Validator

```js
// Pure validation functions
function isValidName(name)     { ... }  // non-empty after trim, max 100 chars
function isValidAmount(amount) { ... }  // numeric, 0.01 – 999,999,999.99
function isValidCategory(cat)  { ... }  // one of ['Food', 'Transport', 'Fun']

// Returns { name: errMsg|null, amount: errMsg|null, category: errMsg|null }
function validateForm(name, amount, category) { ... }
```

## Data Models

### Transaction (in-memory and serialized)

| Field | Type | Constraints | Notes |
|---|---|---|---|
| `id` | string | unique, non-empty | `crypto.randomUUID()`, fallback to `Date.now().toString()` |
| `name` | string | 1–100 chars after trim | trimmed before storage |
| `amount` | number | 0.01 – 999,999,999.99 | stored as float |
| `category` | string | `'Food'` \| `'Transport'` \| `'Fun'` | validated enum |
| `createdAt` | number | positive integer | `Date.now()` at creation |

### LocalStorage Schema

```
Key:   "expense_visualizer_transactions"
Value: JSON.stringify(Transaction[])
```

Example stored value:
```json
[
  { "id": "abc123", "name": "Lunch", "amount": 12.50, "category": "Food", "createdAt": 1700000000000 },
  { "id": "def456", "name": "Bus fare", "amount": 2.00, "category": "Transport", "createdAt": 1700000010000 }
]
```

**Validation on load:** Each parsed item must have all five fields with correct types; any item failing this check causes the entire dataset to be discarded and an error shown.

### Category Aggregation (for chart)

```js
// Computed on every updateChart() call — never stored
{
  Food:      number,  // sum of amounts for Food transactions
  Transport: number,  // sum of amounts for Transport transactions
  Fun:       number   // sum of amounts for Fun transactions
}
```

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Valid transactions are always added to the list

*For any* non-empty item name (1–100 characters after trimming), amount in range [0.01, 999,999,999.99], and valid category (`Food`, `Transport`, `Fun`), submitting the form SHALL result in the transaction list length increasing by exactly one and the new transaction appearing at the top of the list.

**Validates: Requirements 1.3, 1.5, 2.3**

---

### Property 2: Whitespace-only or invalid inputs are always rejected

*For any* item name that is empty or composed entirely of whitespace characters, or any amount outside [0.01, 999,999,999.99] or non-numeric, or any category value not in `{Food, Transport, Fun}`, the form submission SHALL be rejected: the transaction list length SHALL remain unchanged and no new transaction SHALL be written to LocalStorage.

**Validates: Requirements 1.3, 1.4**

---

### Property 3: Balance equals sum of all transaction amounts

*For any* collection of transactions (including the empty collection), the value displayed by Balance_Display SHALL equal the arithmetic sum of all transaction amounts, formatted with a currency symbol and exactly two decimal places.

**Validates: Requirements 4.2, 4.5**

---

### Property 4: Deleting a transaction removes it from the list and balance

*For any* non-empty transaction collection and any transaction in that collection, deleting it SHALL result in: (a) the transaction no longer appearing in the list, (b) the balance decreasing by exactly that transaction's amount, and (c) the transaction not being present in the serialized LocalStorage data.

**Validates: Requirements 3.2, 3.3, 3.5**

---

### Property 5: LocalStorage serialization round-trip preserves transactions

*For any* valid transaction collection, serializing it to LocalStorage as JSON and then reading and parsing it back SHALL produce a collection that is value-equivalent to the original (same transactions with the same id, name, amount, category, and createdAt fields).

**Validates: Requirements 6.1, 6.2, 6.3**

---

### Property 6: Chart slice proportions reflect category totals

*For any* transaction collection with at least one transaction, the proportion of each Category's slice in the pie chart SHALL equal that Category's summed amount divided by the total of all transaction amounts. Categories with a zero total SHALL be excluded from the chart.

**Validates: Requirements 5.1, 5.6**

---

### Property 7: Malformed LocalStorage data produces empty initial state

*For any* value stored in LocalStorage that is not valid JSON, or valid JSON that does not conform to the Transaction schema, initializing the app SHALL produce an empty transaction list, a balance of $0.00, and an empty chart state — and SHALL display an inline error message.

**Validates: Requirements 6.4, 6.5**

## Error Handling

### Validation Errors (Input_Form)
- Displayed as inline `<span class="error-msg">` elements adjacent to each invalid field.
- Cleared on each new submission attempt before re-validating.
- Form submission is blocked; no state mutation occurs.

### LocalStorage Write Failure (on add/delete)
- Caught via `try/catch` around `localStorage.setItem(...)`.
- The in-memory `transactions[]` mutation is **not** reverted — the UI reflects the change.
- An inline error notification is shown: *"Warning: Your change could not be saved. It will be lost when you close this tab."*
- The notification persists until the next successful save or page reload.

### LocalStorage Read Failure (on init)
- Caught via `try/catch` around `localStorage.getItem(...)` and `JSON.parse(...)`.
- Also triggered if any parsed transaction is missing required fields or has wrong types.
- App initializes to empty state.
- Inline error shown: *"Could not load saved transactions. Your previous data may be corrupted."*

### Chart.js CDN Load Failure
- Detected by checking `typeof Chart === 'undefined'` after DOMContentLoaded.
- `<canvas>` is hidden; a visible error message is shown in the chart container: *"Chart unavailable: could not load Chart.js."*
- All other features (form, list, balance, storage) continue to function normally.

### Edge Cases
| Scenario | Behavior |
|---|---|
| Amount with more than 2 decimal places entered | Browser `<input type="number" step="0.01">` and validator reject it |
| Very large number of transactions (e.g., 10,000) | `renderList()` re-renders all; no virtualization — acceptable given client-side single-tab use |
| Duplicate transaction names | Allowed; each transaction has a unique `id` |
| `crypto.randomUUID()` not available (very old browser) | Fallback to `Date.now().toString() + Math.random()` |

## Testing Strategy

### Unit Tests (example-based)

Unit tests target pure logic functions in isolation, using concrete examples and edge cases:

| Function | Test Cases |
|---|---|
| `validateForm()` | Valid inputs pass; empty name rejected; whitespace-only name rejected; amount 0 rejected; amount 999,999,999.99 passes; amount 1,000,000,000 rejected; invalid category rejected |
| `formatCurrency()` | `12.5` → `"$12.50"`; `0` → `"$0.00"`; `1234567.89` → `"$1,234,567.89"` |
| `loadTransactions()` | Empty storage → `[]`; valid JSON → parsed array; malformed JSON → `[]` + error; missing field → `[]` + error |
| `saveTransactions()` | Round-trip: save then load returns equivalent array |
| Category aggregation | Correct sums per category; zero-amount categories excluded |

### Property-Based Tests

Property-based testing is used to validate universal invariants across the range of valid inputs. The recommended library is **fast-check** (loaded via CDN for testing, or used in a simple Node test harness).

Each property test runs a minimum of **100 iterations** with randomized inputs.

| Property | Test Tag |
|---|---|
| Property 1: Valid transactions are always added | `Feature: expense-budget-visualizer, Property 1: Valid transactions are always added` |
| Property 2: Whitespace/invalid inputs are rejected | `Feature: expense-budget-visualizer, Property 2: Whitespace/invalid inputs are rejected` |
| Property 3: Balance equals sum of amounts | `Feature: expense-budget-visualizer, Property 3: Balance equals sum of amounts` |
| Property 4: Deleting removes transaction and updates balance | `Feature: expense-budget-visualizer, Property 4: Deleting removes transaction and updates balance` |
| Property 5: LocalStorage round-trip preserves transactions | `Feature: expense-budget-visualizer, Property 5: LocalStorage round-trip preserves transactions` |
| Property 6: Chart slice proportions reflect category totals | `Feature: expense-budget-visualizer, Property 6: Chart slice proportions reflect category totals` |
| Property 7: Malformed data produces empty initial state | `Feature: expense-budget-visualizer, Property 7: Malformed data produces empty initial state` |

### Integration / Smoke Tests

- Load the app in Chrome, Firefox, Edge, Safari; verify no uncaught JS errors in console.
- Simulate CDN failure (block Chart.js URL via DevTools) and verify graceful degradation.
- Fill LocalStorage with malformed JSON manually and reload; verify error message and empty state.
- Verify WCAG AA contrast ratios using a browser accessibility tool (e.g., axe, Lighthouse).
- Resize viewport from 320px to 1920px and verify no overlapping or horizontal scrollbar.

### Performance Checks
- Lighthouse audit for initial load time on a throttled connection (25 Mbps / 50ms RTT).
- Manually add/delete a transaction while observing DevTools Performance timeline to confirm UI updates within 100ms.
